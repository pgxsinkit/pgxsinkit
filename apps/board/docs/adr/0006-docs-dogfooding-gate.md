# The board build is the toolkit's documentation dogfooding pass

`apps/board` consumes `@pgxsinkit/*` exactly as an external user or AI assistant
would — by installing the packages and reading the docs, not the source. That
makes the build a one-time, high-fidelity test of whether the docs actually let
someone use the libraries the way they're meant to be used. So every build phase
runs a **docs-review gate**:

- While building, reach for the docs first (Starlight content, getting-started,
  the AI-assistant page, the generated API reference) and `llms.txt` **before**
  reading library source. Any time progress required reading source — and a real
  consumer would need the same thing — that is a **documentation gap** (pure
  internals a consumer never touches do not count).
- The phase is not "done" until the gaps it surfaced are closed: update the
  **Starlight content** and/or the **source JSDoc** so a fresh consumer could build
  that phase from the docs alone, then run `bun run --cwd apps/docs build` and
  inspect the regenerated `llms.txt` / `llms-full.txt` for the new material.
- Findings are logged in [`apps/board/docs/consumer-review.md`](../consumer-review.md):
  what was needed, where it was looked for, whether the docs had it, and how it
  resolved. The log is the dogfooding audit trail; the **fixes land in the toolkit
  docs**, not in the board.

`llms.txt` is auto-generated by the `starlight-llms-txt` plugin from the Starlight
content, so the human docs + source JSDoc are the **single source for both
audiences** — we improve those and verify the generated `llms.txt`, never
hand-maintain it.

## Considered Options

- **One docs-review phase at the end of the build** — rejected: findings go stale,
  and you lose the compounding benefit of building each later phase against docs
  that the earlier phases already corrected.

## Consequences

- Scope is bounded to the toolkit surface the board actually uses — documented
  because a real consumer needed it, not for completeness.
- A genuine _ergonomics_ gap (not just a docs gap) is fixed upstream in the
  toolkit, never worked around in the board (per the repo's upstream-first rule).
